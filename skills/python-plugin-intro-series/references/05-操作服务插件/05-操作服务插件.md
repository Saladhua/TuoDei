## 一、单据操作

- 系统内置 **160+** 种操作类型，支持单据菜单、列表菜单、单据体菜单调用
- 预置**空操作**，方便自定义操作封装
- 操作可针对整单（如保存、提交、审核、删除），也可针对单据体分录（新增分录、删除分录、批量填充），或两者兼可（状态转换）
- 操作具备4个特性：
  - **操作编码** — 操作唯一标识
  - **操作名称** — 显示名称
  - **操作类型** — 区分功能类别
  - **操作所属实体** — 绑定到哪个实体的操作
- 可在BOS设计器中查看、编辑单据操作列表

## 二、操作服务插件简介

### 基本概念

- **APP层插件**，基类：`AbstractOperationServicePlugIn`
- 依赖**单据操作**触发，非界面
- 运行机制类似 **"流水线作业"**，事件按顺序执行，**某个事件不会独立触发**
- **事务：**部分环节以事务机制运行，默认超时 **10min**，事务中发生异常支持回滚
- **无 `View` 成员！**

### 整体架构与执行流程

操作服务插件并非孤立运行，它嵌入在从 UI 到数据层的完整操作链中。整个链路由以下4个角色组成：

1. **AbstractDynamicFormOperationExt** — 前端/表单层抽象扩展类
2. **二开表单插件（列表插件）** — 二次开发的表单级插件
3. **AbstractOperationService（app）** — 后端应用层抽象操作服务
4. **二开服务插件（app）** — 二次开发的服务级插件

**执行流程（按顺序）：**

1. **表单层启动（AbstractDynamicFormOperationExt）：**
   - `Execute` — 操作开始执行
   - `DoAuthPermission`（可重写）— 权限校验，失败则中止
   - `StartNetworkCtrl`（可重写）— 开启网络连接/会话
   - `ExecuteOperation`（可重写）— 操作具体实现，通常调用后续服务层

2. **表单插件介入（二开表单插件）：**
   - `BeforeDoOperation` — 操作前预处理
   - `Execute` — 开始执行服务（过渡到服务层）
   - `PrepareProperties`（可重写）— 向操作服务添加常用字段
   - `Validate` — 执行操作配置的校验器

3. **服务层执行（AbstractOperationService）：**
   - `Initialize`（可重写）— 服务初始化
   - `Execute` — 开始执行服务
   - `PrepareProperties`（可重写）— 再次添加/合并字段
   - `Validate` — 服务层校验
   - `ExecuteOperationTransaction`（可重写）— 执行操作事务（数据修改、跨表操作）

4. **服务插件扩展点（二开服务插件）：**
   - `OnPrepareOperationServiceOption` — 选项设置
   - `OnPrepareProperties` — 定制加载指定字段
   - `InitializeOperationResult` — 操作结果初始化
   - `BeforeExecuteOperationTransaction` — 事务执行前处理
   - `BeginOperationTransaction` — 调用操作事务前触发（事务内）
   - `EndOperationTransaction` — 调用操作事务后触发（事务内）
   - `AfterExecuteOperationTransaction` — 事务执行后处理

**设计模式与架构特点：**

- **模板方法模式** — 表单层和服务层都定义了固定执行骨架，可变部分交给子类/开发者重写
- **责任链/事件管道** — 二开插件通过钩子注入自定义逻辑，形成"中间件"处理链
- **分层解耦** — 表现层(权限/网络)、应用服务层(编排/校验/事务)、插件层(二开入口)清晰分离
- **事务边界清晰** — `BeginOperationTransaction`/`ExecuteOperationTransaction`/`EndOperationTransaction` 显式暴露，前后置扩展，适合强数据一致性场景

**典型使用场景：**
- 自定义按钮操作（审核、过账、同步等）
- 复杂业务动作：权限校验 + 网络连接 + 多表事务更新 + 二开多环节插入逻辑

### 使用操作服务插件的优势

1. **触发面广** — 在单据列表和单据维护界面都可触发；表单插件和操作服务插件都能实现的功能，优先选择操作服务插件
2. **代码量低** — 操作服务插件在权限控制和校验之后再执行，80%+ 校验逻辑可通过 BOS 配置实现
3. **外部调用方便** — 支持 WebAPI 和其他插件调用
4. **复用性强** — 同功能可快速复用到多单据、多操作

## 四、操作服务插件事件介绍

### 事件一览表

| 事件 | 事务 | 说明 |
|------|:----:|------|
| `OnPrepareOperationServiceOption` | 外 | 设置是否启动事务、批量处理 |
| `OnPreparePropertys` | 外 | **字段预加载（必须！）** |
| `OnAddValidators` | 外 | 自定义校验器干预 |
| `BeforeExecuteOperationTransaction` | 外 | 操作校验后、实现前预处理 |
| `BeginOperationTransaction` | **内** | 事务开始后、操作执行前 |
| `EndOperationTransaction` | **内** | 操作执行后、逻辑处理 |
| `RollbackData` | 外 | 异常时回滚内存数据 |
| `AfterExecuteOperationTransaction` | 外 | 事务提交后，不影响操作 |

### 事件详解

```python
# 选项设置，在操作服务初始化时执行
# 可设置是否启动事务、是否支持批量处理等
def OnPrepareOperationServiceOption(e):
    e.SupportTransaction = True
    e.SurportBatchTransaction = True


# 字段预加载事件 — 非常重要
# 服务插件默认只加载单据编号、单据ID等关键字段
# 需要读取的其他字段必须在此事件中预加载
# 后续事件取字段报错"字段标识不存在"时，检查是否在此预加载
def OnPreparePropertys(e):
    e.FieldKeys.Add("FBillTypeID")
    e.FieldKeys.Add("字段标识")


# 校验器干预
# 可添加自定义校验器，或取消已有校验器
# 校验不通过则不继续执行
# 仅APP层执行的操作默认执行校验器(保存/提交/审核/删除等)
# Web层操作(下推/选单/复制等)不默认执行，可通过表单插件强制触发
def OnAddValidators(e):
    vlidators = e.Validators


# 执行操作事务前事件（事务外触发）
# 在操作校验之后、操作实现之前执行
# 数据库处理不受操作事务保护
def BeforeExecuteOperationTransaction(e):
    e.Cancel = False
    if (e.Cancel == True):
        e.CancelMessage = ("{0}被取消了！").format(this.FormOperation.OperationName.GetString(2052))


# 操作事务前事件（事务内触发），支持回滚
# 在操作校验之后、操作事务开始之后、操作执行之前
# 数据库处理受操作事务保护
# 可用于数据准备、获取操作执行前数据
# 如审核操作此处单据状态还是"审核中"，删除操作此处是最后一次能从DB取到数据
def BeginOperationTransaction(e):
    for billObj in e.DataEntitys:
        billId = billObj["Id"]
        billNo = billObj["BillNo"]


# 操作事务后事件（事务内触发），支持回滚
# 在操作执行之后，数据库已更新成操作后数据
# 删除操作在此无法通过SQL查询到单据数据
# 适合保证操作正常完成后再处理的任务（如SQL更新相关单据）
def EndOperationTransaction(e):
    for billObj in e.DataEntitys:
        billId = billObj["Id"]
        billNo = billObj["BillNo"]


# 内部事务执行失败后回滚数据事件（事务外触发）
# 仅在操作执行过程中发生异常时触发
def RollbackData(e):
    for billObj in e.DataEntitys:
        billId = billObj["Id"]


# 执行操作事务后事件（事务外触发）
# 在操作执行后、事务提交之后执行
# 数据库处理不受操作事务保护
# 尽量不要在此事件中抛出异常（事务已提交，异常会导致后续业务不正常）
# 正确的做法是把异常信息放到OperationResult中
def AfterExecuteOperationTransaction(e):
    for billObj in e.DataEntitys:
        billId = billObj["Id"]
        billNo = billObj["BillNo"]
    result = OperateResult()
    result.SuccessStatus = True
    result.Message = "提示信息"
    this.OperationResult.OperateResult.Add(result)
```
